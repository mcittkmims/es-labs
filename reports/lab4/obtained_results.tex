\subsubsection{Test 1.1: Relay Toggle ON (Key A)}

Pressing key A on the keypad sends a toggle command to the binary actuator. The serial terminal immediately confirms the command with \texttt{[INPUT] Relay cmd: ON}. After the debounce period (3 consecutive confirmations at 100\,ms intervals = 300\,ms), the relay module activates: the white load LED illuminates, and the LCD updates its first line to show \texttt{Relay:ON}. The 2-second periodic serial report confirms the relay is in the ON state.

\subsubsection{Test 1.3: Relay Debounce Validation}

To verify that the software debounce prevents false toggles, rapid successive presses of key A are performed. Despite multiple key events arriving within a short window, the relay state changes only once the command has been stable for 3 consecutive 100\,ms control cycles. The debounce counter value, visible in the serial report, confirms that transient command changes are filtered out. The relay does not ``chatter'' (rapidly toggling ON and OFF) during fast keypad interaction.

\subsubsection{Test 2.2: Signal Conditioning and Ramping}

After confirming a target of 75\%, the conditioning pipeline processes the command. The saturation stage passes 75 unchanged (within [0, 100]). The median filter requires several cycles to fill its 5-sample window. The EWMA smoothes the transition. Most visibly, the ramping stage limits the output increase to 5\% per 100\,ms cycle, so the blue PWM LED gradually brightens over approximately 1.5 seconds rather than jumping instantly to 75\%.

The serial report shows the pipeline stages: the raw command at 75.0\%, the conditioned (post-EWMA) value converging toward 75\%, and the ramped output incrementing in 5\% steps (0, 5, 10, 15, ... 70, 75). The PWM register value correspondingly increases from 0 to 191 (75\% of 255).

\subsubsection{Test 2.3: Overload Alert (PWM $>$ 90\%)}

Setting the PWM target to 95\% (via B, 9, 5, \#) triggers the overload detection sequence. As the ramped output climbs past 90\%, the ThresholdAlert FSM transitions from NORMAL to DEBOUNCE\_HIGH. After 3 consecutive readings above 90\% (300\,ms), the FSM enters the ALERT state. The red LED turns on, the green LED turns off, and the LCD displays \texttt{!ALERT} on the second line. \subsubsection{Test 2.4: Alert Hysteresis and Clearing}

To verify hysteresis behavior, the PWM target is reduced to 85\% (within the hysteresis band of 80--90\%). The alert remains active because the FSM only clears when the value drops \emph{below} the low threshold of 80\%. Setting the target to 70\% causes the ramped output to decrease past 80\%, triggering the DEBOUNCE\_LOW state. After 3 consecutive sub-80\% readings, the FSM returns to NORMAL: the green LED turns back on, the red LED turns off, and the LCD shows \texttt{OK} again. This confirms that the hysteresis band prevents alert oscillation near the threshold boundary.

\subsection{System Observations}

The system demonstrates responsive command processing, with keypad input detected within 50\,ms of the key press. The relay debounce adds a 300\,ms delay before the physical relay state changes, which is appropriate for preventing false toggles from noisy membrane contacts. The PWM ramping at 5\% per 100\,ms produces visibly smooth LED brightness transitions, which would translate to gentle motor speed changes in a real application.

The structured serial reports, printed every 2 seconds (or on demand via key D), provide comprehensive diagnostic information including the raw command, conditioned value, ramped output, PWM register value, and alert state. This level of observability would be valuable for tuning the conditioning parameters in a production system.

Memory utilization remains well within the Arduino Mega 2560's constraints at 25.7\% RAM and 9.8\% Flash, suggesting the architecture could support significantly more complex applications on the same hardware platform. The FreeRTOS scheduler manages the three tasks without observable timing jitter, and no deadlock or data corruption was encountered during testing.
