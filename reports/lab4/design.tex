
\section{Design}

\subsection{System Architecture Diagrams}

\subsubsection{Component-Level Architecture}

The system integrates six hardware components around the Arduino Mega 2560 microcontroller. The keypad provides user input through a matrix GPIO interface, while the LCD receives status updates via I2C. The relay module is controlled by a single digital GPIO pin, and the PWM actuator (simulated by a blue LED) receives an analog-equivalent signal on a PWM-capable pin. Two status LEDs provide immediate visual feedback: green for normal operation and red for overload alert conditions.

\subsubsection{Layered System Architecture}

The software is organized into five layers following the AUTOSAR-inspired embedded systems architecture pattern. Each layer communicates only with its immediate neighbors, enforcing clean separation of concerns and enabling module reuse across labs.

\caption{Layered software architecture of the dual-actuator control system}
\label{fig:layered_architecture}

The APP layer contains the three FreeRTOS tasks and the lab entry point. The SRV layer provides the ActuatorConditioner (which wraps the SignalConditioner with ramping logic) and the SharedState module (mutex-protected data exchange). The ECAL layer implements the SignalConditioner pipeline and ThresholdAlert FSM as reusable, hardware-independent modules. The MCAL layer contains all hardware drivers that directly call Arduino GPIO, I2C, and PWM functions. The HW layer represents the physical components.

\subsection{Part 1 --- Binary Actuator Control (Relay)}

This subsection covers the design aspects specific to the binary actuator: the relay-driven load controlled via keypad toggle commands with software debouncing.

\subsubsection{Binary Actuator Data Flow}

The binary actuator control path is straightforward: the user presses key A on the keypad, which sets a boolean command flag in the shared state. The control task reads this flag, applies software debouncing (3 consecutive confirmations at 100\,ms intervals), and drives the relay GPIO pin accordingly. The relay module switches the load circuit through its Normally Open contact.

\subsubsection{Relay Debounce Algorithm}

The debounce algorithm prevents false relay toggles caused by noisy keypad contacts or accidental double-presses. Rather than debouncing the keypad itself (which the Keypad library already handles), this debounce operates on the \emph{command signal} to ensure that the relay only changes state after the command has been stable for a configurable number of control cycles.

The debounce threshold is set to 3 confirmations. Since the control task runs every 100\,ms, the relay state changes 300\,ms after a stable command is received. This delay is imperceptible for human-initiated control while being effective at filtering transient command changes.

\subsubsection{Binary Actuator Electrical Schematic}

The relay module is connected to pin D7 of the Arduino Mega. When the pin is driven HIGH, the optocoupler inside the module activates the electromagnetic coil, switching the COM terminal from NC to NO. The load LED is connected between COM and ground, with 5V supplied through the NC terminal (which connects to COM when the relay is energized).

\subsection{Part 2 --- Analog Actuator Control (PWM)}

This subsection covers the design aspects specific to the analog actuator: the PWM-driven load controlled via keypad numeric entry with a four-stage signal conditioning pipeline and overload alerting.

\subsubsection{Analog Actuator Data Flow}

The analog actuator control path is more complex than the binary path. The user enters a target percentage (0--100) using keys B (enter mode), 0--9 (digits), and \# (confirm). The raw command value passes through a four-stage conditioning pipeline before reaching the PWM output. An overload alert FSM monitors the conditioned output and controls the status LEDs.

\subsubsection{Signal Conditioning Pipeline Flowchart}

The conditioning pipeline processes each raw command value through four sequential stages. The saturation stage clamps the input to the physical actuator limits (0--100\%). The median filter maintains a sliding window of the 5 most recent values and selects the middle one, removing impulsive outliers. The EWMA stage applies exponential smoothing with $\alpha = 0.4$, balancing responsiveness and stability. Finally, the ramping stage limits the output change to 5\% per 100\,ms cycle, ensuring gradual actuator transitions.

\subsubsection{Overload Alert FSM}

The overload detection uses a 4-state hysteresis FSM with debounce confirmation. An alert triggers when the conditioned PWM output exceeds 90\% for 3 consecutive evaluations, and clears when it drops below 80\% for 3 consecutive evaluations. The hysteresis band (80\%--90\%) prevents the alert from oscillating when the actuator operates near the threshold boundary.

\begin{figure}[H]
\centering
\begin{tikzpicture}[>=Stealth, auto, node distance=3.5cm,
        every state/.style={draw, circle, minimum size=1.5cm, font=\scriptsize}]
    \node[state, initial]   (normal)  {NORMAL};
    \node[state]            (debH)    [right of=normal]  {DEB\_H};
    \node[state]            (alert)   [right of=debH]    {ALERT};
    \node[state]            (debL)    [below of=debH]    {DEB\_L};
    \path[->]
        (normal) edge              node[font=\tiny] {val $>$ 90\%}    (debH)
        (debH)   edge              node[font=\tiny] {3 confirms}      (alert)
        (debH)   edge [bend right] node[font=\tiny, below] {val $\leq$ 90\%} (normal)
        (alert)  edge              node[font=\tiny, right] {val $<$ 80\%}     (debL)
        (debL)   edge              node[font=\tiny] {3 confirms}      (normal)
        (debL)   edge [bend right] node[font=\tiny, right] {val $\geq$ 80\%} (alert);
\end{tikzpicture}
\caption{Overload alert FSM state diagram with hysteresis}
\label{fig:overload_fsm}
\end{figure}

When the FSM enters the ALERT state, the red LED turns on and the green LED turns off. The LCD display shows \texttt{!ALERT} to indicate the overload condition. Reducing the PWM target below 80\% initiates a transition through the DEBOUNCE\_LOW state, requiring 3 consecutive sub-threshold readings before returning to NORMAL and restoring the green LED.

\subsubsection{Analog Actuator Electrical Schematic}

The PWM output on pin D3 drives a blue LED through a 220\,$\Omega$ current-limiting resistor. The LED brightness varies proportionally to the duty cycle, visually representing the analog actuator output level. In a real application, the same PWM signal would drive a motor controller (e.g., L298N H-bridge) or a proportional valve driver.

\subsection{Electrical Schematics}

\subsubsection{Component Specification}

The circuit uses the following components with their electrical specifications:

\begin{itemize}
    \item \textbf{Arduino Mega 2560} --- MCU board, 5\,V logic, 16\,MHz, 54 digital I/O pins
    \item \textbf{Relay module} --- 5\,V single-channel, optocoupler-isolated, NO/NC/COM terminals
    \item \textbf{LCD 1602 I2C} --- 16x2 character display, I2C address 0x27, 5\,V supply
    \item \textbf{4x4 Keypad} --- membrane matrix, 8 GPIO lines (D22--D29)
    \item \textbf{Blue LED} --- PWM actuator indicator, $V_f \approx 3.0$\,V, with 220\,$\Omega$ resistor
    \item \textbf{Green LED} --- system normal indicator, $V_f \approx 2.1$\,V, with 220\,$\Omega$ resistor

\subsection{Modular Implementation}

\subsubsection{MCAL Layer: Relay Driver (Binary Actuator)}

The Relay module provides a simple interface for controlling a relay connected to a digital GPIO pin. It supports configurable active level (HIGH or LOW) and tracks the current relay state internally.

\begin{lstlisting}[language=C++, caption=Relay driver interface (Relay.h), label=lst:relay_h]
class Relay {
public:
    Relay(uint8_t pin, bool activeHigh = true);
    void init();
    void turnOn();
    void turnOff();
    void toggle();
    void setState(bool on);
    bool isOn() const;
private:
    uint8_t _pin;
    bool    _activeHigh;
    bool    _state;
};
\end{lstlisting}

The implementation maps the logical ON/OFF state to the physical pin level based on the \texttt{activeHigh} configuration. When \texttt{activeHigh} is true, turning ON writes HIGH to the pin; when false, it writes LOW. This accommodates both active-HIGH relay modules (common in Arduino ecosystem) and active-LOW modules (common in industrial applications).

\subsubsection{MCAL Layer: PWM Actuator Driver (Analog Actuator)}

The PwmActuator module abstracts PWM output control, accepting a duty cycle percentage (0--100\%) and mapping it to the 8-bit range required by \texttt{analogWrite()}.

\begin{lstlisting}[language=C++, caption=PwmActuator driver interface (PwmActuator.h), label=lst:pwm_actuator_h]
class PwmActuator {
public:
    PwmActuator(uint8_t pin);
    void init();
    void setDuty(float duty);
    float getDuty() const;
    uint8_t getRawPwm() const;
private:
    uint8_t _pin;
    float   _duty;
};
\end{lstlisting}

The \texttt{setDuty()} method clamps the input to [0, 100] before computing \texttt{pwmVal = duty * 255 / 100} and calling \texttt{analogWrite()}. This percentage-based interface decouples the application logic from the hardware-specific 8-bit range.

\subsubsection{SRV Layer: Actuator Conditioner (Analog Actuator)}

The ActuatorConditioner module extends the existing SignalConditioner with a ramping stage. It wraps the three-stage pipeline (saturation, median filter, EWMA) and adds gradual output transitions to protect mechanical actuators from sudden changes.

\begin{lstlisting}[language=C++, caption=ActuatorConditioner interface (ActuatorConditioner.h), label=lst:act_cond_h]
class ActuatorConditioner {
public:
    ActuatorConditioner(uint8_t medianWindow, float ewmaAlpha,
                        float minClamp, float maxClamp,
                        float maxRampStep);
    float process(float rawCommand);
    float getRampedOutput() const;
    float getConditionedTarget() const;
    const SignalConditioner& getSignalConditioner() const;
    void reset();
};
\end{lstlisting}

The ramping logic computes the difference between the conditioned target and the current output, then limits the change to \texttt{maxRampStep} per call. This ensures that even if the user commands a jump from 0\% to 100\%, the actual output ramps up at 5\% per 100\,ms cycle, reaching full speed in approximately 2 seconds.

\subsubsection{ECAL Layer: Threshold Alert FSM (Analog Actuator)}

The ThresholdAlert module, originally developed in Lab 3 for temperature monitoring, is reused here to detect overload conditions on the analog actuator. The module implements a 4-state hysteresis FSM with configurable high/low thresholds and debounce confirmation count. For this lab, the thresholds are set to 90\% (trigger) and 80\% (clear) with 3 required confirmations.

\subsubsection{APP Layer: Shared State Module}

The SharedState module defines the \texttt{ActuatorState} structure containing all data exchanged between the three tasks. It includes fields for both actuator subsystems: relay command/actual state with debounce counter (binary), and PWM command/conditioned/ramped values with raw register output (analog). A one-shot \texttt{reportRequested} flag allows the input task to trigger an immediate serial report via the display task. A FreeRTOS mutex protects concurrent access.

\begin{lstlisting}[language=C++, caption=Shared state structure (shared\_state.h), label=lst:shared_state_h]
struct ActuatorState {
    // Binary actuator
    bool relayCommandOn;
    bool relayActualOn;
    uint8_t relayDebounceCount;
    // Analog actuator
    float pwmCommandPercent;
    float pwmConditioned;
    float pwmRamped;
    uint8_t pwmRawValue;
    // Alert and control
    bool overloadAlert;
    bool inputModeAnalog;
    char inputBuffer[4];
    uint8_t inputBufferLen;
    bool reportRequested;
};
\end{lstlisting}

\subsubsection{APP Layer: FreeRTOS Tasks}

Three tasks run concurrently, each with a dedicated period and priority. The Input task (priority 3, 50\,ms) has the highest priority to ensure responsive keypad scanning. The Control task (priority 2, 100\,ms) processes both binary and analog actuator commands and drives all hardware outputs. The Display task (priority 1, 500\,ms) updates the LCD and prints serial reports at a relaxed rate to avoid consuming excessive CPU time with I/O operations.

All tasks follow the same pattern: acquire the mutex, read/write the shared state, release the mutex, then perform hardware I/O outside the critical section where possible. This minimizes mutex hold time and reduces the risk of priority inversion.
