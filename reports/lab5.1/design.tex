\subsection{Block Diagrams}

\subsubsection{Runtime Data Flow}

The runtime data flow is split around a mutex-protected shared state object. The input and acquisition tasks are the only tasks that produce user and sensor inputs. The control task consumes the latest sample and setpoint, then writes the desired relay command. The actuation task is the only task that touches the relay driver, and the display task periodically snapshots the state for LCD and Serial Plotter output. This makes the timing model easy to reason about because hardware writes are not scattered across unrelated modules.

\subsubsection{Task Synchronization Sequence}

The acquisition, control, and actuation tasks are connected through binary semaphores, while the shared state is protected by a mutex. The acquisition task gives a new-sample semaphore after writing a new DHT22 and potentiometer sample. The control task blocks on that semaphore, updates the hysteresis output, and gives an actuator semaphore. The relay task blocks on the actuator semaphore and is therefore woken only when a fresh control command is ready. The display task does not block the control path; it simply locks the mutex briefly every 500 ms to copy a snapshot.

\subsubsection{Component Specification}

\begin{itemize}
    \item Arduino Mega 2560: ATmega2560, 5 V logic, 16 MHz.
    \item DHT22: digital temperature and humidity sensor connected to D2.
    \item Potentiometer: analog setpoint source connected to A0.
    \item Keypad: 4x4 matrix, rows on D11--D8 and columns on D7--D4.
    \item LCD 1602 I2C: address \(0x27\), connected through SDA/SCL.
    \item Relay module: logic input on D12, active-high in the provided wiring reference.

\subsubsection{Circuit Connections}

The DHT22 VCC and GND pins are connected to the Arduino 5 V and GND rails, and its data line is wired to D2. The potentiometer is wired as a voltage divider from 5 V to GND with its wiper on A0. The keypad row pins are D11, D10, D9, and D8, while the column pins are D7, D6, D5, and D4. The LCD I2C backpack uses the Mega SDA and SCL pins. The relay module logic side is powered from 5 V and controlled by D12. The relay contacts switch the external LED load branch, which is powered separately.

\subsection{Modular Implementation}

\subsubsection{Implementation Overview}

The implementation follows the same separation that was modeled in the architecture diagrams. Lab-specific behavior is located in `labs/lab/lab5\_1`, while reusable hardware and control abstractions are located under `labs/lib`. The application does not call `digitalWrite`, `analogRead`, DHT protocol methods, or LCD library methods directly from the control algorithm. Instead, the low-level operations are hidden behind wrappers such as `DhtSensor`, `AnalogSetpointInput`, `Relay`, `KeypadInput`, and `LcdDisplay`. This keeps the control task focused on the question ``what should the actuator do?'' and leaves the driver modules responsible for the question ``how is the hardware accessed?''

The task design also follows the extra automatic-control requirements from the laboratory guide. Acquisition, control, actuation, display, and input are separated into dedicated modules. The shared variables are protected by a FreeRTOS mutex, and the control path is coordinated through binary semaphores so a new control decision is produced only after a fresh acquisition cycle. The keypad module uses the Keypad library debounce mechanism, while the control algorithm itself uses hysteresis to suppress relay oscillations around the setpoint.

\subsubsection{Application Layer: Lab Entry Point}

The lab entry point is intentionally small. It initializes the STDIO serial bridge at 9600 baud, prints a startup banner that documents the keypad commands and pin mapping, initializes the shared state, and creates the five tasks. After setup completes, `lab5\_1Loop()` remains empty because the Arduino FreeRTOS integration transfers execution to the scheduler. This preserves the repository rule that the lab-specific entry point orchestrates the application but does not contain low-level driver logic.

\subsubsection{Shared State and Synchronization}

The shared state module is the communication contract between tasks. It stores the most recent measured temperature, measured humidity, potentiometer raw value, potentiometer setpoint, manual setpoint, active setpoint source, hysteresis band, computed thresholds, desired relay command, actual relay state, and diagnostic counters. A mutex, `xLab5StateMutex`, protects this structure so that multi-field updates are seen consistently by the display and control tasks.

Two semaphores express the event order of the control chain. The new-sample semaphore is given by the acquisition task after it writes a sample. The control task waits on this semaphore, so the hysteresis calculation cannot run on stale data in a tight loop. After calculating the new command, the control task gives the actuator semaphore, which wakes the actuation task. This separation is useful on real hardware because the relay driver remains centralized in one task, while the control task remains pure decision logic.

\begin{lstlisting}[language=C++, caption=Shared state fields used by the control path, label=lst:shared_state_fields]
struct Lab5ControlState {
    float measuredTempC;
    float activeSetpointC;
    float hysteresisBandC;
    float lowerThresholdC;
    float upperThresholdC;
    bool sensorValid;
    bool controlCommandOn;
    bool actuatorOn;
};
\end{lstlisting}

\subsubsection{Service Layer: Hysteresis Controller}

The `OnOffHysteresisController` module implements the core control law. It stores the selected setpoint, the full hysteresis band, and the previous output state. The previous output state is important: if the measured temperature is inside the hysteresis band, the controller must not compute a new command from scratch. It must preserve its last command, otherwise the relay would still chatter around the reference.

For a heating-style actuator, the lower threshold represents the point where heat is needed, and the upper threshold represents the point where the actuator can stop. In this lab, the relay is turned ON when the measured temperature is less than or equal to the lower threshold and turned OFF when the measured temperature is greater than or equal to the upper threshold. If the sensor read is invalid, the control task calls `forceOutput(false)` before updating the shared command, which makes the relay fail safe.

\begin{lstlisting}[language=C++, caption=ON-OFF hysteresis controller interface, label=lst:hysteresis_header]
class OnOffHysteresisController {
public:
    OnOffHysteresisController(float setpoint, float hysteresisBand);
    void init();
    void setConfig(float setpoint, float hysteresisBand);
    bool update(float measuredValue);
    void forceOutput(bool outputOn);
    bool isOutputOn() const;
    float getLowerThreshold() const;
    float getUpperThreshold() const;
};
\end{lstlisting}

\begin{lstlisting}[language=C++, caption=Hysteresis state update, label=lst:hysteresis_update]
bool OnOffHysteresisController::update(float measuredValue) {
    if (_state == HYSTERESIS_OUTPUT_OFF &&
        measuredValue <= getLowerThreshold()) {
        _state = HYSTERESIS_OUTPUT_ON;
    } else if (_state == HYSTERESIS_OUTPUT_ON &&
               measuredValue >= getUpperThreshold()) {
        _state = HYSTERESIS_OUTPUT_OFF;
    }
    return isOutputOn();
}
\end{lstlisting}

\subsubsection{ECAL Layer: Acquisition Wrappers}

The `DhtSensor` wrapper owns DHT22 protocol access and validity checking. It calls the Adafruit DHT library, rejects `NaN` readings, and exposes the last valid temperature and humidity through simple getters. This means the acquisition task does not need to know the details of the DHT timing protocol or library error values.

The `AnalogSetpointInput` wrapper owns ADC acquisition and maps the potentiometer reading to the 15--35 degrees Celsius setpoint range. The conversion is linear: ADC value 0 maps to the minimum setpoint and ADC value 1023 maps to the maximum setpoint. This wrapper makes the setpoint range easy to change from configuration without touching the task logic.

\begin{lstlisting}[language=C++, caption=Acquisition task sequence, label=lst:acquisition_task]
bool sensorOk = s_dht.read();
float temperature = s_dht.getTemperatureC();
float humidity = s_dht.getHumidityPercent();
float potSetpoint = s_setpointPot.readValue();

lab5StateLock();
state->sensorValid = sensorOk;
state->measuredTempC = temperature;
state->measuredHumidityPercent = humidity;
state->potSetpointC = potSetpoint;
state->activeSetpointC =
    state->setpointSource == SETPOINT_SOURCE_POT
        ? potSetpoint
        : state->manualSetpointC;
lab5StateUnlock();
\end{lstlisting}

\subsubsection{Input Task}

The input task scans the keypad every 50 ms. The `KeypadInput` wrapper configures a 20 ms debounce interval, so the task receives stable key events rather than raw electrical transitions. Key `A` toggles between potentiometer and manual setpoint modes. Keys `B` and `C` decrement and increment the manual setpoint by 0.5 degrees Celsius. Key `D` cycles the hysteresis band from 0.5 to 5.0 degrees Celsius. Digit keys start numeric setpoint entry, `\#` confirms the typed value, and `*` cancels the edit.

The task writes only user-intent fields in shared state. It does not calculate thresholds, access the relay, or write to the LCD. That keeps interaction policy independent from control and display policy.

\subsubsection{Acquisition Task}

The acquisition task runs every 2000 ms, matching the practical sampling rate of the DHT22. Each cycle reads both sensor values and the potentiometer setpoint before taking the mutex. This is deliberate: slow I/O is performed outside the critical section, and the mutex is held only while copying values into shared state. If the selected setpoint source is the potentiometer, the active setpoint is updated from A0. If the selected source is manual, the active setpoint is copied from `manualSetpointC` instead.

\subsubsection{Control and Actuation Tasks}

The control task is event-driven. It waits for a new sample, copies the current temperature, active setpoint, and hysteresis band, then updates the hysteresis controller outside the mutex. Once the command is calculated, it writes the command, controller state, lower threshold, upper threshold, and diagnostic counter back into shared state. It then gives the actuator semaphore.

The actuation task owns the `Relay` instance. This is a small but important design choice because relay writes are centralized in exactly one task. The task wakes only when the control task gives the actuator semaphore, reads the desired command, applies it through the relay wrapper, and writes back the actual relay state. It also increments `actuatorSwitches` whenever the physical output changes, which can later be used to evaluate relay wear or controller aggressiveness.

\begin{lstlisting}[language=C++, caption=Control-to-actuation handoff, label=lst:control_actuation_handoff]
state->controlCommandOn = commandOn;
state->lowerThresholdC = s_controller.getLowerThreshold();
state->upperThresholdC = s_controller.getUpperThreshold();
state->controlCycles++;
xSemaphoreGive(xLab5ActuatorSemaphore);
\end{lstlisting}

\subsubsection{Display and Serial Plotter Task}

The display task runs every 500 ms. It copies the shared state while holding the mutex briefly, then formats the LCD and serial output outside the critical section. This prevents slow display operations from blocking acquisition or control. The LCD alternates between two pages: a primary page with measured temperature, setpoint, relay state, and sensor status; and a diagnostics page with hysteresis band, humidity, setpoint source, and lower/upper thresholds. During numeric setpoint entry, the LCD switches to an edit page so the typed digits are visible before confirmation.

The Serial Plotter output is intentionally compact and stable. Each line uses named fields: `SetPoint`, `Value`, `Output`, `Low`, `High`, and `Valid`. Arduino Serial Plotter can display these named channels over time, which directly satisfies the requirement to transmit SetPoint, Value, and Output for real-time analysis.

\begin{lstlisting}[language=C++, caption=Serial Plotter report format, label=lst:plotter_output]
printf("SetPoint:%s Value:%s Output:%u Low:%s High:%s Valid:%u\r\n",
       plotSetpoint,
       plotValue,
       snapshot.actuatorOn ? 1U : 0U,
       plotLow,
       plotHigh,
       snapshot.sensorValid ? 1U : 0U);
\end{lstlisting}

\subsubsection{Timing Configuration and Justification}

The selected recurrences are chosen around the hardware limitations and human interface needs. The keypad scan period is 50 ms because human button presses are slow compared with MCU execution and because the wrapper already debounces key transitions. The DHT22 acquisition period is 2000 ms to avoid invalid reads caused by excessive polling. The display period is 500 ms, fast enough to feel responsive during keypad entry but slow enough to avoid unnecessary I2C traffic. Control and actuation are event-driven rather than periodic, so the relay decision follows the newest sample without wasting CPU time between samples.
