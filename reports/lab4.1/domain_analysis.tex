\section{Domain Analysis}

\subsection{Objective of the Laboratory Work}

The objective of this laboratory work is to design and implement a binary actuator control system on the Arduino Mega 2560 microcontroller that accepts commands from the user via a serial terminal, applies a multi-stage signal conditioning pipeline to eliminate spurious transitions, and drives a physical output while continuously reporting the system state on both an LCD display and the serial monitor.

The core learning objectives are to understand the concepts of binary actuator control, software-based signal conditioning for discrete signals (saturation, debouncing, and persistent state validation), and preemptive task scheduling with FreeRTOS. The lab also reinforces the principle of separating concerns across modular software layers, from raw input parsing through signal processing to hardware actuation and reporting.

\subsection{Problem Definition}

The task, as specified by the laboratory instructor, is reproduced below:

\begin{quote}
\textit{Control Actuator Binar}

\textit{Pentru un actuator binar (ex.: bec de iluminare prin releu sau alt actuator convenit cu mentorul), se va realiza o aplicație pentru MCU care:}
\begin{itemize}
    \item \textit{interpretează comenzi de control de la utilizator prin STDIO (serial sau keypad)}
    \item \textit{controlează actuatorul}
    \item \textit{afișează starea curentă pe un display LCD}
\end{itemize}

\textit{Aplicația va implementa:}
\begin{itemize}
    \item \textbf{Control actuator binar:} Primește comenzi ON/OFF prin STDIO. Aplică debouncing. Controlează starea la o recurență configurabilă (50–100 ms). Expune starea printr-o interfață internă.
    \item \textbf{Condiționare semnal:} Aplică saturare, debouncing software, și validare persistentă a stării.
    \item \textbf{Afișare și raportare:} Afișează pe LCD sau STDIO un raport structurat la 500 ms.
\end{itemize}
\end{quote}

The system therefore requires three distinct processing stages. The user sends the string \texttt{ON} or \texttt{OFF} via the serial terminal. That raw command passes through a conditioning pipeline that first saturates the value to a strict boolean, then requires the value to remain stable for a minimum debounce window, and finally confirms it with a minimum number of consecutive identical samples before the hardware output is actually toggled. The conditioned state is displayed on the LCD and reported periodically on the serial monitor.

\subsection{Used Technologies}

\subsubsection{UART Serial Communication and STDIO Redirection}

UART (Universal Asynchronous Receiver-Transmitter) is the hardware serial interface used to connect the Arduino Mega 2560 to the host PC. The microcontroller's UART0 peripheral is connected to a USB-to-serial converter, making it accessible as a virtual COM port on the host. The application configures the interface at 9600 baud with the standard 8N1 framing (8 data bits, no parity, 1 stop bit).

Rather than using the Arduino-specific \texttt{Serial.print()} API at the application level, the \texttt{StdioSerial} library redirects the C standard I/O streams — \texttt{stdout} and \texttt{stdin} — to the hardware serial port using the AVR libc \texttt{fdev\_setup\_stream()} mechanism. This allows all formatted output to use the portable \texttt{printf()} function and all input to be read through \texttt{Serial.available()} / \texttt{Serial.read()}, keeping the application layer independent of hardware-specific serial APIs.

\subsubsection{Binary Actuator Control and Signal Conditioning}

A binary actuator is any device that operates in exactly two discrete states: active (ON) and inactive (OFF). Common examples include relays driving mains loads, solenoid valves, indicator LEDs, and buzzers. In this laboratory the actuator is simulated by a red LED connected to a current-limiting resistor on pin D7. The LED is chosen as a safe, visible proxy for a relay — the software interface is identical regardless of the actual load, since the \texttt{Led} driver abstracts the hardware behind \texttt{turnOn()}/\texttt{turnOff()} calls that could be replaced with relay driver logic without modifying any application code.

Control of a binary actuator from a user interface introduces the risk of spurious state changes caused by rapid repeated commands, communication noise, or accidental double-presses. In an industrial context, rapid cycling of relay contacts (known as "chattering") dramatically reduces the relay's mechanical lifetime and can cause arcing across the contacts, potentially damaging downstream equipment. Even in low-power applications, unnecessary state transitions waste energy and generate electromagnetic interference. A signal conditioning pipeline addresses these risks by introducing three filtering stages applied in sequence.

\paragraph{Stage 1 — Saturation.} The first stage maps any raw input value to a strict element of the boolean set $\{$\texttt{false}, \texttt{true}$\}$. While this is trivially satisfied for a serial command parser that already outputs a boolean, saturation becomes essential when the input source is an analog sensor, a noisy digital line, or a multi-valued protocol message. By placing saturation at the pipeline entry, the subsequent stages are guaranteed to operate on clean boolean values regardless of the input source.

\paragraph{Stage 2 — Software Debouncing.} The second stage prevents the actuator state from changing unless the input has been stable for a minimum duration (configured as 100~ms in this laboratory). This mirrors the hardware debouncing technique used for mechanical pushbuttons and relay contacts, where the physical switch bounces between open and closed states for several milliseconds after activation. Software debouncing uses a timer that resets every time the input changes; only when the timer exceeds the debounce threshold is the input considered stable. The debounce period must be chosen to exceed the longest expected transient duration while remaining short enough to preserve acceptable response latency.

\paragraph{Stage 3 — Persistent State Validation.} The third stage requires the input to be confirmed by a minimum number of consecutive identical samples (configured as 3 confirmations in this laboratory). This provides an additional layer of filtering beyond time-based debouncing: even if a transient glitch is long enough to survive the debounce window, it must appear consistently across multiple sampling intervals to be accepted. The dual-guard design — requiring \emph{both} the debounce period and the persistence count to be satisfied simultaneously — ensures robust noise rejection suitable for safety-critical applications as described in IEC~61508.

\subsubsection{FreeRTOS Preemptive Task Scheduler}

FreeRTOS is a real-time operating system kernel that provides preemptive multitasking on resource-constrained microcontrollers. The kernel maintains a ready queue of tasks sorted by priority, and the scheduler always dispatches the highest-priority ready task. When a higher-priority task becomes ready (for example, because its delay period has elapsed), the kernel immediately preempts the currently running task, saves its context (register file and stack pointer) to the task control block (TCB), and restores the context of the higher-priority task. This preemption happens transparently via a timer interrupt — on the AVR ATmega2560, the kernel uses Timer 3 configured at a default tick rate of 200~Hz (5~ms per tick).

On the Arduino Mega 2560, the \texttt{feilipu/FreeRTOS} library ports the kernel to the AVR architecture. Because the ATmega2560 has only 8~KB of SRAM and each task requires its own stack, careful stack sizing is critical. The \texttt{portSTACK\_TYPE} on AVR is \texttt{uint8\_t}, meaning stack sizes specified in the \texttt{xTaskCreate()} call are in bytes, not words. A task that uses \texttt{printf()} requires at least 200--300 bytes of stack space for the formatting buffer alone, plus additional space for local variables, function call frames, and the saved register context (32 general-purpose registers plus SREG and the program counter on AVR). In this laboratory, the input task is allocated 256 bytes, the actuator task 256 bytes, and the display task 512 bytes (to accommodate the larger \texttt{printf()} formatting calls in the periodic STDIO report).

Each task runs a while-forever loop and uses \texttt{vTaskDelayUntil()} rather than \texttt{vTaskDelay()} for period control. The difference is significant: \texttt{vTaskDelay()} delays for a specified number of ticks \emph{from the moment the call is made}, which means the actual period includes the task's execution time and varies with processing load. In contrast, \texttt{vTaskDelayUntil()} delays until an absolute tick count is reached, compensating for the execution time and producing a consistent period regardless of how long the task body takes to execute. This is essential for the actuator control task, which must run at a predictable 75~ms recurrence to satisfy the conditioning pipeline's timing assumptions.

Preemptive scheduling ensures that higher-priority tasks (input reading at priority 3) can interrupt lower-priority ones (display at priority 1), guaranteeing responsive user interaction regardless of display rendering time.

\paragraph{Mutex-Based Synchronization.}
When multiple tasks share data, a synchronization mechanism is required to prevent race conditions — situations where one task reads a partially-updated structure written by another. FreeRTOS provides binary semaphores and mutexes for this purpose. A mutex (mutual exclusion semaphore) additionally implements priority inheritance: if a low-priority task holds the mutex and a high-priority task attempts to take it, the kernel temporarily raises the holder's priority to match the waiter's, preventing unbounded priority inversion. In this laboratory, the \texttt{ActuatorShared} structure is protected by a single FreeRTOS mutex. Each task acquires the mutex with a 5~ms timeout, performs its read or write operation, and immediately releases it. The critical sections are kept deliberately short — typically a single field assignment — to minimize the time any task blocks waiting for access.

\subsubsection{I2C Protocol and LCD Display}

The Inter-Integrated Circuit (I2C) protocol is a two-wire synchronous serial bus originally developed by Philips Semiconductor (now NXP) for short-distance intra-board communication. The bus uses two open-drain signal lines — SDA (Serial Data) and SCL (Serial Clock) — both pulled high by external resistors. Communication follows a master-slave architecture: the master (the Arduino Mega 2560 in this case) generates the clock and initiates transactions, while the slave (the PCF8574 I/O expander on the LCD backpack) responds with acknowledgements and data.

Each I2C transaction begins with a START condition (SDA falling while SCL is high), followed by a 7-bit slave address and a read/write direction bit. The addressed slave acknowledges by pulling SDA low during the ninth clock cycle. Data bytes follow, each acknowledged individually, until the master issues a STOP condition (SDA rising while SCL is high). The Arduino Mega 2560 provides a dedicated hardware I2C peripheral (TWI — Two-Wire Interface) on pins SDA (D20) and SCL (D21), which handles the low-level bit timing and acknowledgement detection in hardware, freeing the CPU for other tasks.

The LCD module used in this laboratory is a 16$\times$2 character display based on the Hitachi HD44780 controller. The HD44780 natively requires an 8-bit or 4-bit parallel data interface plus three control signals (RS, RW, E), which would consume 7--11 GPIO pins on the microcontroller. The PCF8574 I2C backpack solves this by converting the two-wire I2C bus into the parallel interface required by the HD44780, reducing the pin count to just SDA and SCL. The standard I2C address of 0x27 is used for the LCD module; this address is set by the A0--A2 address pins on the PCF8574, which are all pulled high by default.

The \texttt{LcdDisplay} library wraps the third-party \texttt{LiquidCrystal\_I2C} driver, providing a simplified line-based API. The \texttt{showTwoLines()} method writes both LCD lines in a single call, automatically padding shorter strings with spaces to clear residual characters from the previous update. This prevents visual artifacts where old characters remain visible when the new string is shorter than the previous one — a common issue when directly using the \texttt{setCursor()}/\texttt{print()} API without explicit clearing.

\subsection{Hardware Components}

\subsubsection{Arduino Mega 2560}

The Arduino Mega 2560 is a microcontroller development board built around the 8-bit AVR ATmega2560 running at 16 MHz. It provides 256 KB of Flash program memory, 8 KB of SRAM for runtime data, 54 digital I/O pins, and 16 analog inputs. Four hardware UART ports enable simultaneous serial communication with the host PC and other peripherals. The on-board USB-to-serial converter allows programming and monitoring over a standard USB cable. In this laboratory the board acts as the system's central processing unit, executing the FreeRTOS kernel and all three application tasks.

\subsubsection{LED and Current-Limiting Resistor}

A standard 5 mm red LED is used to simulate the binary actuator output. An LED requires a current-limiting resistor in series to prevent excessive current from damaging the device. With a supply of 5 V, a typical LED forward voltage of 2.0 V, and a desired current of approximately 14 mA, the required resistance is $(5.0 - 2.0) / 0.014 \approx 214\,\Omega$, so a standard 220 $\Omega$ resistor is used. In a real-world application the LED and resistor would be replaced by a relay module driving a mains load; the software interface and signal conditioning pipeline remain identical regardless of the actual load type.

\subsection{System Architecture and Justification}

The system follows a layered hardware-abstraction architecture divided into five tiers. The hardware tier consists of the physical GPIO pin, resistor, and LED. The MCAL (Microcontroller Abstraction Layer) tier is represented by the \texttt{Led} driver, which encapsulates all \texttt{pinMode()} and \texttt{digitalWrite()} calls behind a stable API. The ECAL (ECU Abstraction Layer) tier contains the \texttt{BinaryConditioner}, which implements the signal processing logic independent of any specific hardware. The service tier is represented by \texttt{ActuatorControl}, which combines the conditioner with the LED driver to form a self-contained actuator module. The application tier consists of the three FreeRTOS tasks and the shared state structure that orchestrate the overall system behaviour.

FreeRTOS is chosen over a simple cooperative scheduler because it provides true preemption and priority enforcement — the input reader task at priority 3 will always preempt the display task at priority 1, ensuring that serial input is never missed regardless of how long the display update takes. The mutex on the shared state structure prevents partially-written structs from being read by another task mid-update.

\subsection{Case Study — Relevant Applicability of the Solution}

Binary actuator control with signal conditioning is ubiquitous in industrial automation. A programmable logic controller (PLC) controlling a conveyor-belt drive motor, a solenoid valve in a pneumatic system, or a contactor in a motor control centre all employ debounce and persistence logic to prevent spurious state transitions from noisy digital inputs or momentary command spikes. The same pattern appears in building management systems where light relays and HVAC damper actuators must not cycle rapidly in response to transient occupancy sensor readings, since relay contact wear increases with switching frequency.

In consumer electronics, smart home relay modules use software debouncing to prevent a user's voice command from being applied multiple times due to repeated speech recognition events. In automotive body-control modules, door-lock actuators are controlled through validated discrete inputs to prevent uncommanded locking caused by electromagnetic interference on vehicle wiring harnesses. The laboratory's three-stage pipeline — saturation, debounce, persistence — directly mirrors the input processing found in safety-critical embedded systems described by the ISO 26262 and IEC 61508 functional-safety standards.
