Each layer communicates only with the layer immediately above or below it through well-defined interfaces declared in header files. The \texttt{Led} driver isolates all \texttt{pinMode()} and \texttt{digitalWrite()} calls; the \texttt{BinaryConditioner} works purely on boolean values with no hardware knowledge; the \texttt{ActuatorControl} service wires those together; and the application tasks orchestrate timing and inter-task communication.

\subsection{Electrical Schematics}

\subsubsection{Component Specification}

The circuit uses three components beyond the Arduino Mega 2560 itself. The red LED on pin D7 serves as the binary actuator output; its anode is connected to D7 through a series 220\,\si{\ohm} current-limiting resistor, and its cathode to GND. The 16x2 LCD module with PCF8574 I2C backpack is powered from the 5 V rail and communicates via the I2C bus on pins D20 (SDA) and D21 (SCL). The PCF8574 I2C address is 0x27. No pull-up resistors are required for the I2C lines as they are provided internally by the LCD backpack module.

\subsection{Modular Implementation}

\subsubsection{MCAL Layer: Led Driver}

The \texttt{Led} driver provides a stable GPIO abstraction. It wraps a single pin identifier and tracks the current state in a \texttt{bool} field, so callers never need to use \texttt{digitalRead()} to query the state. The \texttt{init()} method configures the pin as an output and guarantees an initial OFF state, preventing the LED from flickering at boot.

\begin{lstlisting}[language=C++, caption=Led driver interface (\texttt{Led.h} excerpt), label=lst:led_h]
class Led {
public:
    Led(uint8_t pin);
    void init();      // Set OUTPUT, initial OFF
    void turnOn();    // Set HIGH
    void turnOff();   // Set LOW
    void toggle();
    bool isOn() const;
private:
    uint8_t ledPin;
    bool    state;
};
\end{lstlisting}

\subsubsection{ECAL Layer: BinaryConditioner}

The \texttt{BinaryConditioner} implements the signal processing pipeline. It operates entirely on boolean values, making it hardware-independent. The \texttt{process()} method is called once per actuator task tick with the current raw command.

\begin{lstlisting}[language=C++, caption=BinaryConditioner key logic (\texttt{BinaryConditioner.cpp}), label=lst:conditioner_cpp]
bool BinaryConditioner::process(bool rawInput) {
    bool saturated = rawInput ? true : false;  // Stage 1: Saturation
    _rawState = saturated;

    if (saturated != _pendingState) {
        // New value: restart validation
        _pendingState    = saturated;
        _debounceStartMs = millis();
        _confirmCount    = 1;
    } else if (_pendingState != _conditionedState) {
        _confirmCount++;  // Stage 3: accumulate confirmations
    }

    if (_pendingState != _conditionedState) {
        _inDebounce = true;
        uint32_t elapsed = millis() - _debounceStartMs;
        // Stage 2+3: accept only when BOTH conditions satisfied
        if (_confirmCount >= _persistCount && elapsed >= _debouncePeriodMs) {
            _conditionedState = _pendingState;
            _inDebounce       = false;
            _confirmCount     = 0;
        }
    } else {
        _inDebounce = false;
    }
    return _conditionedState;
}
\end{lstlisting}

The key design choice is that stages 2 (debounce) and 3 (persistence) must both be satisfied simultaneously before a transition is committed. Requiring only the persistence count would fail against very rapid repeated commands; requiring only the debounce period would fail against periodic glitches spaced further apart than the window.

\subsubsection{Service Layer: ActuatorControl}

\texttt{ActuatorControl} is the service-layer module that wires the \texttt{BinaryConditioner} to the \texttt{Led} driver. Its \texttt{update()} method feeds the current raw command into the conditioner pipeline and drives the LED output only when the conditioned state changes. This avoids unnecessary \texttt{digitalWrite()} calls on every tick.

\begin{lstlisting}[language=C++, caption=ActuatorControl::update() implementation, label=lst:actuator_update]
void ActuatorControl::update() {
    bool rawBool     = (_rawCommand == State::ON);
    bool conditioned = _conditioner.process(rawBool);

    State newState = conditioned ? State::ON : State::OFF;
    if (newState != _currentState) {
        _currentState = newState;
        if (_currentState == State::ON) {
            _led.turnOn();
        } else {
            _led.turnOff();
        }
    }
}
\end{lstlisting}

\subsubsection{Application Layer: FreeRTOS Tasks and Shared State}

The three tasks communicate through \texttt{g\_actuator}, a global \texttt{ActuatorShared} struct protected by a FreeRTOS mutex. Every read or write to the shared fields is surrounded by \texttt{xSemaphoreTake()} / \texttt{xSemaphoreGive()} calls with a 5–10 ms timeout. The short timeout ensures that if a task somehow stalls, other tasks are not blocked indefinitely.

\begin{lstlisting}[language=C++, caption=Shared state structure (\texttt{actuator\_data.h}), label=lst:shared_state]
typedef struct {
    ActuatorControl::State pendingCmd;     // from input task
    ActuatorControl::State actuatorState;  // from actuator task
    bool                   isDebouncing;
    uint8_t                confirmCount;
    uint32_t               toggleCount;
    SemaphoreHandle_t      mutex;
} ActuatorShared;
\end{lstlisting}

The input reader task polls \texttt{Serial.available()} every 10 ms rather than blocking on \texttt{fgets()}. This non-blocking approach is essential in FreeRTOS: a blocking read would consume the task's CPU slice while waiting, preventing lower-priority tasks from running under some scheduler configurations. The 10 ms polling granularity is more than adequate for human-speed terminal input.
