The Lab 4.1 firmware was compiled successfully using the PlatformIO \texttt{lab4\_1} environment. The build system detected all seven library dependencies automatically via the \texttt{deep+} Library Dependency Finder mode: FreeRTOS, LiquidCrystal\_I2C, ActuatorControl, BinaryConditioner, Led, StdioSerial, LcdDisplay, and the Arduino Wire library for I2C. The resulting firmware fits comfortably within the hardware limits of the ATmega2560: RAM usage is 18.9\,\% (1546 bytes out of 8192 bytes) and Flash usage is 7.1\,\% (18024 bytes out of 253952 bytes), leaving ample headroom for future extension.

\subsection{Command Interface Validation}

\subsubsection{Test: ON Command with Conditioning Delay}

The first test verifies that typing \texttt{ON} in the serial terminal is correctly parsed, accepted, and eventually applied to the actuator after the conditioning pipeline is satisfied. Because the debounce window is 100 ms and the actuator task runs at 75 ms, the transition is committed after the second or third task tick following the command. During this interval the STDIO report shows ``Cond: DEBOUNCING'' and ``Confirm: 1/3'' or ``2/3''. Once confirmed, the report shows ``State: ON'', ``Cond: STABLE'', and the LED turns on in the Wokwi simulation.

\subsection{System Observations}

The most notable observation during testing is the visible delay between issuing a command and the actuator responding. With a 100 ms debounce window and a 75 ms task period, the earliest a state change can be committed is approximately 150–225 ms after the command is first received. This is intentional and demonstrates the conditioning pipeline working correctly. The delay would be imperceptible for a light relay application (where a sub-second response is acceptable) but would need to be reduced for time-critical control loops.

The LCD correctly reflects the conditioned state rather than the raw command. When the user types ON but the conditioning has not yet been satisfied, the LCD continues to show OFF with DEBOUNCING on the second line. This provides clear visual feedback that the system has received the request and is in the process of validating it. The STDIO report adds numerical detail by showing the confirmation count progress.

RAM usage of 1546 bytes (18.9\,\%) with three FreeRTOS tasks, two library instances, and a mutex leaves 6646 bytes free. This is consistent with the relatively small per-task stack sizes (192–320 words) chosen for this application. If additional complexity such as a keypad input or multiple actuators were added, stack sizes would need to be reviewed.

\subsection{Verification Summary}

\begin{itemize}
    \item \textbf{Compilation:} Firmware compiles successfully for the \texttt{lab4\_1} environment. RAM 18.9\,\%, Flash 7.1\,\%.
    \item \textbf{Initialization:} Startup banner prints correctly; LCD and serial are both initialized before FreeRTOS tasks start.
    \item \textbf{Signal Conditioning:} All three stages (saturation, debounce, persistence) verified by observing the confirmation count and DEBOUNCING status during command transitions.
    \item \textbf{Command Parsing:} ON and OFF commands accepted; unknown input rejected with error message; backspace editing works correctly in the terminal.
    \item \textbf{Display:} LCD line 1 tracks conditioned state; line 2 tracks conditioning status. Periodic STDIO reports contain all required fields.
    \item \textbf{Architecture:} All hardware calls confined to library modules; application tasks contain no direct \texttt{digitalWrite()} or \texttt{analogRead()} calls.
\end{itemize}
